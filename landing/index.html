The problem

Your agents are good at the next 30 minutes. Bad at the next 30 days.

The model isn't the issue. The plumbing is. Sessions reset, knowledge evaporates with the conversation, and your CLAUDE.md slowly becomes a junk drawer.

⤴

Sessions reset

Compaction, new conversation, new machine. Yesterday's reasoning chain is gone before you sat down today.

✦

Knowledge dies in chat

An agent writes a perfect runbook in the conversation — and then the conversation closes. The runbook isn't anywhere you can fetch it.

≈

Vector stores hide why

"Semantically similar" isn't good enough. You can't audit retrieval, can't force a specific note in, can't tell why one chunk beat another.

The substrate

A directory, a CLI, an MCP server. That's the whole thing.

Bundles are markdown files with YAML frontmatter. The CLI gives you ergonomics; the MCP server gives your agents access. Both are optional adapters over a directory of plain files.

📁

Files on disk are source-of-truth

Every bundle is a markdown file under ~/.substrate/bundles/YYYY-MM-DD/. Open them in any editor. Grep them. Diff them. Substrate adds nothing you can't inspect.

⎇

Git is the history

Every add and edit auto-commits. Your audit trail is git log. Your diff is git diff. No proprietary versioning.

⌥

MCP-native

Five tools: list_bundles, get_bundle, search_bundles, get_by_date, log_use. Wire it into any MCP-speaking agent in one config line.

◉

Falsifiable retrieval

Every fetch logs to a TSV. The dashboard surfaces which bundles actually earn their keep — and which are shelf-ware ready for deletion.

A bundle, end-to-end

Write it once. Pull it from any agent.

You author bundles in calm time. Your agents fetch them in busy time. This is the entire pattern.

# Migration playbook

You are about to run an Alembic migration against production.

## Pre-checks

1. Confirm the migration is in `alembic/versions/` and head matches.
2. If it touches a table > 1M rows, escalate.
3. Run `alembic upgrade head --sql` and review the output.

## Execution

4. `SET statement_timeout = '60s'` for the session.
5. Run staging first, verify row counts and FKs.
6. Only then run production.

## Done when

- `alembic current` shows the new head.
- Smoke test query returns expected rows.
- No errors in app log within 60 seconds.
# Author the bundle once
$ substrate add "prompt run migration" --tag prompt,playbook,migration
# your $EDITOR opens; write the playbook; save; substrate commits

# Find it later
$ substrate search migration

Use cases

The four patterns that pay off.

Substrate is the smallest layer that solves all four. Pick one to start; the others compound.

01 — HANDOFF

Cross-session continuity

Write what was decided and what's half-done before stopping. Tomorrow's agent reads it and resumes — no re-explaining, no lost reasoning.

02 — PROMPT

Pre-authored prompts

Write a careful playbook once in calm time. The agent picks it up later in busy time and follows it verbatim. Your calmer self leaves instructions for your rushed self.

03 — CONVENTION

Shared knowledge pool

Run three agents on one repo. They all search the same convention bundles. Update once, every agent stays coherent on its next search.

04 — GOTCHA

Debugging breadcrumbs

Bug took three hours? Write the root cause once. Six days later, when a related issue surfaces, the agent finds the bundle in 80ms instead of re-spelunking.

For agents

One config line. Five MCP tools.

Substrate ships a stdio MCP server. Add one entry to your editor's MCP config and your agent gets list_bundles, get_bundle, search_bundles, get_by_date, and log_use.

That's the entire surface. Read by id, search by tag, fetch by date, log what was used. Authoring stays CLI-only on purpose — agents read; humans write.
